## 1. Bugs & Issues

### BUG-01 · HIGH · Server boundary violation in client component

**File:** `src/lib/components/feedback/FeedbackForm.svelte:7`

```ts
import type { FeedbackType, FeedbackSeverity } from '$lib/server/db/types';
```

A client-side `.svelte` component imports from `$lib/server/`. SvelteKit's static analysis enforces the `$lib/server/` boundary regardless of whether the import is a `type`-only import; in some build configurations this triggers a hard build error. The imported types (`FeedbackType = 'bug' | 'feature' | 'other'` and `FeedbackSeverity = 'low' | 'medium' | 'high' | 'blocker'`) are plain string-union types that have no reason to live behind the server guard.

**Suggested fix:** Move `FeedbackType` and `FeedbackSeverity` to `src/lib/contracts/feedback.ts` (or `src/lib/shared/`). Update the import in `FeedbackForm.svelte`, `src/routes/api/feedback/validation.ts`, and `src/lib/server/db/index.ts`.

### BUG-02 · HIGH · `<html lang>` never reflects the active locale

**Files:** `src/app.html:2`, `src/routes/[lang]/+layout.svelte:17-19`

`app.html` hardcodes `<html lang="en">`. The `[lang]/+layout.svelte` calls `setLocale(lang)` in `$effect` but never updates `document.documentElement.lang`. As a result, screen readers, browser spell-check, and search-engine crawlers always see the page as English — even on `/fr/*` routes. This is an accessibility failure (WCAG 3.1.1 Language of Page — Level A).

**Suggested fix (two parts):**

1. In `src/hooks.server.ts`, add `transformPageChunk` to the `resolve` call:
   ```ts
   return resolve(event, {
     transformPageChunk: ({ html }) => {
       const lang = event.url.pathname.startsWith('/fr') ? 'fr' : 'en';
       return html.replace('lang="en"', `lang="${lang}"`);
     }
   });
   ```
2. In `src/routes/[lang]/+layout.svelte`, inside the existing `$effect`:
   ```ts
   $effect(() => {
     if (lang) {
       setLocale(lang);
       document.documentElement.lang = lang;
     }
   });
   ```

SVG `<text>` elements do not inherit CSS custom properties for `font-size` in all browsers; the workaround is to declare them as inline CSS variables on the SVG element and reference them inside. The current hardcoded values will require manual updates if the visual design changes.

**Suggested fix:** Add CSS variables to the SVG root (e.g. `style="--svg-label-sm: 9px; --svg-label-md: 11px;"`) driven by a theme token or a prop, then reference those variables inside the SVG style block.

### IDEA-01 · Fixed-Asset Depreciation Method Comparator

**Title (EN):** Fixed-Asset Depreciation Methods  
**Title (FR):** Méthodes d'Amortissement des Immobilisations

**Target learner:** Licence 2, Licence 3

**Pedagogical objective:** Students routinely conflate loan amortization (which the existing playground covers) with fixed-asset depreciation, and they rarely see side-by-side how the choice of method (straight-line, declining-balance, units of production) changes the annual charge, book value curve, and cumulative deferred tax. The legacy `depreciation-playground.html` exists but is not yet a native Svelte module.

**Core interaction:** Enter one asset (cost, useful life, residual value, estimated production units). Three parallel schedules render side-by-side (linéaire / dégressif SYSCOHADA / unités d'œuvre). A shared chart overlays the three book-value curves. A KPI strip shows the Year-1 charge, total accumulated depreciation, and the resulting impact on net income for each method.

**Minimum viable inputs/outputs:**
- Inputs: acquisition cost, useful life (years), residual value, annual production units (optional), declining-balance coefficient
- Outputs: 3 × annual schedule table, overlaid book-value line chart, comparative KPI strip, SYSCOHADA journal entries for Year 1 (account 281x / 68x)

**Why it complements existing playgrounds:** The amortization playground is financial (loan repayment / PMT logic). This is accounting (asset lifecycle). Together they cover both capital structure and fixed-asset management — two distinct Licence 2 exam topics.

### IDEA-02 · VAT / TVA Accounting Workbench

**Title (EN):** VAT Accounting — OHADA Zone  
**Title (FR):** Comptabilité de la TVA — Zone OHADA

**Target learner:** Licence 1, Licence 2

**Pedagogical objective:** Students in OHADA-zone countries must record TVA collectée (output VAT) and TVA déductible (input VAT) and compute the monthly TVA à reverser. The misconception to fix is treating VAT as an expense rather than a pass-through liability. Most curricula present this theoretically; learners need hands-on entry practice.

**Core interaction:** A transaction list (sales + purchases) with amounts and VAT-applicable flags. The learner enters the debit/credit sides. Real-time feedback shows: TVA collectée, TVA déductible, solde net (TVA à reverser or crédit TVA), and the automatically generated TVA declaration summary (formulaire de déclaration mensuelle). Country picker drives the applicable rate (Cameroun 19.25%, Côte d'Ivoire 18%, Sénégal 18%).

**Minimum viable inputs/outputs:**
- Inputs: transaction list (type, HT amount, TVA applicable Y/N), country
- Outputs: journal entries (accounts 4431/4452), TVA declaration recap, running net TVA balance

**Why it complements existing playgrounds:** No current playground touches indirect taxation. VAT is the highest-frequency compliance task for OHADA-zone SMEs and appears on every Licence 2 and DCG-equivalent exam.

### IDEA-03 · Financial Ratio Analysis Dashboard

**Title (EN):** Financial Ratio Analysis  
**Title (FR):** Analyse par Ratios Financiers

**Target learner:** Licence 3, Master 1

**Pedagogical objective:** Students can compute individual ratios but struggle to interpret them in context — trend, peer comparison, and the interplay between liquidity, solvency, and profitability ratios. The CVP playground covers cost structure; no playground covers balance sheet analysis.

**Core interaction:** Load a pre-built fictional SYSCOHADA company's three-year financial statements (bilan + compte de résultat). Students select which ratios to compute (from a categorised library: liquidité générale, ratio d'endettement, rentabilité des capitaux propres, rotation des stocks, etc.). A dashboard shows each ratio, its formula derivation, a three-year sparkline, and a colour-coded comparison to OHADA-zone sector benchmarks. A "What-if" mode lets students adjust one balance sheet line and see the cascade effect on all ratios.

**Minimum viable inputs/outputs:**
- Inputs: pre-loaded or editable 3-year bilan + compte de résultat
- Outputs: ratio dashboard, sparkline trends, benchmark bands, sensitivity panel

**Why it complements existing playgrounds:** Bridges the gap between journal-entry (transaction recording) and strategic analysis. Directly targets Licence 3 / Master 1 "Analyse financière" courses required by OHADA-zone universities.

### IDEA-04 · OHADA-Zone Payroll Calculator

**Title (EN):** Payroll Computation — OHADA Zone  
**Title (FR):** Calcul de la Paie — Zone OHADA

**Target learner:** Licence 3, Master, practicing professional

**Pedagogical objective:** Payroll computation under OHADA-zone regimes (CNPS, IUTS/IRPP, family allowances) is high-stakes and poorly covered by textbooks. Students confuse gross salary, taxable salary, social base, and net-to-pay. Professionals use error-prone spreadsheets.

**Core interaction:** Enter one employee's gross monthly salary and select a country (Cameroun, Côte d'Ivoire, Sénégal, Gabon). The playground renders a fully itemised payslip: employee deductions (CNPS rate, IRPP/IUTS progressive bracket), employer charges (CNPS patronal, accident-du-travail, allocations familiales), and net-to-pay. A second panel shows the corresponding journal entries (account 641 / 431 / 432 / 444).

**Minimum viable inputs/outputs:**
- Inputs: gross salary, country, number of dependants (for family quotient)
- Outputs: itemised payslip, employer cost summary, journal entries

**Why it complements existing playgrounds:** No current playground touches payroll or social charges. This directly targets two audience segments: Licence 3 students studying "Gestion des Ressources Humaines" and HR/accounting professionals who are an explicit B2B2C segment of OhadaLearn.

### IDEA-05 · Working Capital & Cash Conversion Cycle

**Title (EN):** Working Capital & Cash Conversion Cycle  
**Title (FR):** Besoin en Fonds de Roulement et Cycle de Trésorerie

**Target learner:** Licence 3, Master 1

**Pedagogical objective:** BFR (Besoin en Fonds de Roulement) is systematically misunderstood: students confuse it with cash balance and cannot explain why a profitable company can still face a liquidity crisis. The cash conversion cycle (CCC) — Days Sales Outstanding + Days Inventory Outstanding − Days Payables Outstanding — makes this concrete.

**Core interaction:** Enter a company's receivables, inventory, and payables alongside annual revenue and COGS. The playground computes DSO, DIO, DPO, CCC, and FR (Fonds de Roulement). A Sankey-style flow diagram shows how cash is tied up at each stage of the operating cycle. A "What-if" panel lets students reduce DSO or extend DPO and see the resulting cash freed up.

**Minimum viable inputs/outputs:**
- Inputs: trade receivables, inventory, trade payables, annual revenue, COGS
- Outputs: DSO / DIO / DPO / CCC / BFR / FR, flow diagram, cash-freed sensitivity

**Why it complements existing playgrounds:** TVM and interest cover time-value math; ratio playground (IDEA-03) covers static snapshot analysis. The BFR playground covers the dynamic operating cycle — a distinct Licence 3 exam topic not addressed anywhere else.
