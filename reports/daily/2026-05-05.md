## 1. Bugs & Issues

### BUG-01 — Undefined CSS token with hardcoded fallback
**Severity:** Medium
**File:** `src/lib/components/playground/KpiStrip.svelte:76`

```css
color: var(--orange, #f59e0b);
```

`--orange` is never defined in `src/lib/theme/tokens.ts` (which defines `amber` and `amberGlow`, but not `orange`). When the CSS variable is missing the browser silently falls through to the hardcoded `#f59e0b`, violating the "no hardcoded hex colors" rule.

**Fix:** Add `orange` to `ThemeTokens.colors` in `tokens.ts`, emit it via `css-generator.ts`, and remove the fallback literal. Alternatively, reuse `var(--amber)` if the semantic intent is the same.

---

### BUG-02 — Hardcoded `#fff` in component stylesheet
**Severity:** Medium
**File:** `src/lib/playgrounds/journal-entry/components/EntryHistory.svelte:400`

```css
.btn-confirm {
    background: var(--error);
    border: 1px solid var(--error);
    color: #fff;   /* ← hardcoded */
}
```

The surrounding rules correctly use `var(--error)`. The text color `#fff` should follow the same pattern.

**Fix:** Replace `#fff` with `var(--text-primary)` or introduce a dedicated `--color-on-error` token for text rendered on error-colored backgrounds.

### BUG-03 — API error strings are English-only (i18n gap)
**Severity:** High
**Files:**
- `src/routes/api/waitlist/+server.ts:33,50,63,66`
- `src/routes/api/feedback/+server.ts:40,47,75,78,85`

All server-side JSON error payloads are hardcoded English strings:

```ts
// waitlist/+server.ts
return json({ error: 'Invalid email address' }, { status: 400 });
return json({ error: 'This email is already on the waitlist' }, { status: 409 });
return json({ error: 'Failed to save' }, { status: 500 });
return json({ error: 'Too many requests. Please try again later.' }, { status: 429 });

// feedback/+server.ts
return json({ error: 'Expected multipart/form-data' }, { status: 400 });
return json({ error: 'Screenshot exceeds 5 MB limit' }, { status: 400 });
return json({ error: 'Screenshot must be PNG, JPEG, or WebP' }, { status: 400 });
return json({ error: 'Failed to upload screenshot' }, { status: 500 });
```

Users with a French locale preference will receive English error messages regardless of their setting, breaking the bilingual contract declared in CLAUDE.md.

**Fix (preferred):** Return machine-readable error codes (`{ error: 'ERR_EMAIL_INVALID' }`) and translate them client-side using the i18n system, where locale context is available. This also avoids baking presentation strings into the API layer.

### BUG-04 — Hardcoded spacing and font-size values outside tokens
**Severity:** Low
**Files:** `src/lib/components/Nav.svelte`, `src/lib/components/Footer.svelte`, `src/lib/components/LanguageToggle.svelte`

Multiple raw numeric values appear in component `<style>` blocks, bypassing the token system:

| File | Example violations |
|------|--------------------|
| `Nav.svelte:68,100,108,128,131,155,177,221,240` | `padding: 0 1.5rem`, `font-size: 0.75rem`, `padding: 6px 12px`, etc. |
| `Footer.svelte:30,47,52,64,75` | `padding: 2.5rem 1.5rem 2rem`, `font-size: 1rem`, etc. |
| `LanguageToggle.svelte:46,51,54` | `padding: 2px`, `font-size: 0.72rem`, `padding: 4px 10px` |

These are not bugs that break functionality but they undermine the promise of "change one file to restyle everything." Any future theme change must hunt through component files individually.

**Fix:** Define semantic spacing and typography scale tokens in `tokens.ts` (e.g., `space.xs`, `space.sm`, `text.sm`, `text.base`) and reference them via `var(--space-sm)` etc. in component stylesheets.

### BUG-05 — Static playground HTML files are visually decoupled from the token system
**Severity:** Low
**Files:** `static/playgrounds/*.html`

The four standalone HTML playground files (`amortization-playground.html`, `cvp-playground.html`, `depreciation-playground.html`, `journal-entry-playground.html`) are served as iframes. Inspection of `amortization-playground.html` shows **353 hardcoded hex colors** vs. 122 `var(--)` references. Because these files are self-contained HTML, they cannot import from `tokens.ts` or the SvelteKit build pipeline.

Any theme update in `tokens.ts` leaves the iframe content visually inconsistent — different accent shades, different text colors — creating a broken UI seam visible to the user.

**Fix:** Create a script (e.g., `scripts/sync-playground-tokens.ts`) that reads `tokens.ts` and emits a `:root { ... }` CSS block injected into each HTML file at build time. This maintains the single-source-of-truth principle without requiring a full bundler pipeline for the static files.

### IDEA-01 — VAT & Tax Cascade Calculator
**EN:** VAT & Tax Cascade Calculator
**FR:** Calculateur de TVA et cascade fiscale

**Target learner:** Licence 2–3, practicing accountant

**Pedagogical objective:** Many OHADA-zone students conflate the tax base (assiette fiscale) with the invoice total, and struggle with cascading taxes (TVA + customs duties + excise) applied sequentially. This playground makes the arithmetic and order of application visible and interactive.

**Core interaction:** The learner builds an invoice line by line, selecting which taxes apply (TVA at 18%/19.25%, customs duty %, excise rate) and choosing cascade vs. flat application order. The UI shows the cumulative tax base at each step and the final TTC (toutes taxes comprises) amount.

**Minimum viable inputs:** pre-tax unit price, quantity, list of applicable taxes with rates, cascade toggle.
**Outputs:** tax base at each step, individual tax amounts, total HT, total TTC, tax recoverability indicator (for deductible vs. non-deductible VAT).

**Why it complements existing playgrounds:** CVP and journal-entry playgrounds assume prices are already set; this fills the gap upstream at the procurement/sales invoice level, which is where SYSCOHADA class 4 (receivables/payables) entries originate.

### IDEA-02 — Financial Ratios Analyzer
**EN:** Financial Ratios Analyzer
**FR:** Analyseur de ratios financiers

**Target learner:** Licence 3, Master 1

**Pedagogical objective:** Students can extract raw figures from financial statements but misinterpret what a ratio tells them in isolation vs. in context (industry norms, trend over time). The playground forces them to reason about direction and magnitude, not just calculation.

**Core interaction:** The learner enters a simplified balance sheet and income statement (pre-filled with a SYSCOHADA-structured template). The playground computes 10–12 ratios across four families (liquidity, solvency, profitability, asset efficiency), color-codes them against OHADA-sector benchmarks, and lets the learner manipulate individual line items to see which ratios are most sensitive to which inputs.

**Minimum viable inputs:** current assets, current liabilities, total assets, total equity, revenue, EBIT, net income, working capital, inventories.
**Outputs:** ratio values with formula display, benchmark comparison gauge, sensitivity table showing delta per 10% change in each input.

**Why it complements existing playgrounds:** The journal-entry playground ends at the financial statement level; this playground starts there and teaches the analytical step that follows, completing the full reporting cycle.

### IDEA-03 — Period-End Accruals & Deferrals (Régularisations)
**EN:** Period-End Accruals & Deferrals
**FR:** Régularisations de fin d'exercice (CCA / AAR / FNP / PCA)

**Target learner:** Licence 2, Licence 3

**Pedagogical objective:** The four period-end regularization entries (charges constatées d'avance, produits à recevoir, charges à payer, produits constatés d'avance) are a leading source of errors in SYSCOHADA exams because students confuse which period the economic event belongs to. A visual timeline makes the cut-off principle concrete.

**Core interaction:** The learner is given a scenario (e.g., an insurance premium paid in October covering October–March). A horizontal timeline shows the fiscal year boundary. The learner drags the cost portion to the correct period, selects the appropriate SYSCOHADA regularization account (486, 487, 408, 418), and the playground validates the entry and shows the reversing entry at the start of next year.

**Minimum viable inputs:** transaction date, amount, contract start/end dates, expense or revenue type.
**Outputs:** correct regularization amount, SYSCOHADA account codes, journal entry with date, reversing entry.

**Why it complements existing playgrounds:** The journal-entry playground covers routine transactions; accruals/deferrals are a distinct, exam-heavy topic that requires cut-off reasoning — something none of the current playgrounds address.

### IDEA-04 — Payroll & Social Charges Builder (Fiche de Paie OHADA)
**EN:** Payroll & Social Charges Builder
**FR:** Constructeur de fiche de paie et charges sociales

**Target learner:** Licence 3, Master, practicing HR-accountant

**Pedagogical objective:** Payroll accounting in OHADA-zone countries involves employer vs. employee social charge splits that differ by country (Cameroon, Côte d'Ivoire, Senegal, etc.). Students confuse gross, taxable base, net-to-pay, and total employer cost — four different figures derived from one salary.

**Core interaction:** The learner selects a country (from a short list of major OHADA members), enters the gross salary, and the playground builds a live payslip: it applies the correct social charge rates (CNPS/CNAM/IPRES rates per country), calculates income tax (IRPP/IGR), and derives net-to-pay. A parallel panel shows the full accounting entry (SYSCOHADA class 6 charges + class 4 payables).

**Minimum viable inputs:** gross salary, country, employment category (cadre/non-cadre), allowances (transport, housing).
**Outputs:** payslip table (gross → deductions → net), employer total cost, journal entry with SYSCOHADA account codes.

**Why it complements existing playgrounds:** No existing playground touches SYSCOHADA class 6 (operating expenses) or the payroll cycle, which is a mandatory topic in Licence 3 and professional certification exams.

### IDEA-05 — Working Capital & Cash Cycle Simulator
**EN:** Working Capital & Cash Cycle Simulator
**FR:** Simulateur de BFR et cycle de trésorerie

**Target learner:** Master 1, practicing financial controller

**Pedagogical objective:** BFR (besoin en fonds de roulement) is widely taught as a formula but poorly understood as a dynamic. Students don't intuit how extending customer payment terms or reducing supplier terms can push a profitable business into a liquidity crisis — common in OHADA-zone SMEs.

**Core interaction:** The learner sets a business scenario (daily revenue, cost of goods, inventory days, customer days outstanding, supplier days payable). The playground renders a cash conversion cycle diagram and a 12-week cash flow bar chart. Sliders for each parameter update the diagram in real time, making the causal chain between operational decisions and treasury position immediately visible.

**Minimum viable inputs:** daily revenue, COGS ratio, inventory holding days, customer credit days, supplier payment days, opening cash balance.
**Outputs:** BFR value and sign, cash conversion cycle in days, 12-week treasury bar chart, break-even cash balance needed.

**Why it complements existing playgrounds:** The TVM playground handles discounting; the CVP playground handles breakeven. Neither addresses operational cash management, which is the most common financial pain point for OHADA-zone SMEs and a key topic in management accounting curricula.
