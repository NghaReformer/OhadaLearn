## 1. Bugs & Issues

### BUG-01 · Missing theme tokens referenced everywhere via CSS fallbacks
**Severity:** HIGH
**Files (representative):**
- `src/lib/components/playground/KpiStrip.svelte:76`
- `src/lib/playgrounds/amortization/components/ScheduleTable.svelte:240,246,259,265`
- `src/lib/playgrounds/amortization/components/InputsPanel.svelte:604–605`
- `src/lib/playgrounds/bank-reconciliation/components/BankStatementPanel.svelte:149,175,208–222`
- `src/lib/playgrounds/bank-reconciliation/components/TransactionJournal.svelte:271–394`
- `src/lib/playgrounds/bank-reconciliation/components/LedgerPanel.svelte:149,175,208–222`
- `src/lib/playgrounds/bank-reconciliation/components/ReconciliationStatement.svelte:275–298`
- `src/lib/playgrounds/bank-reconciliation/components/UnmatchedItemsPanel.svelte:188`

**Explanation:** Four semantic color tokens — `--orange`, `--blue`, `--red`, and partial re-uses of `--green` — are referenced widely via `var(--token, #hardcoded-fallback)` but are **not defined** in `src/lib/theme/tokens.ts`. Because the CSS variable never resolves, the hex fallback is permanently active, bypassing the theme system entirely. Any future theme override or dark/light variant will silently not affect these elements.

**Suggested fix:** Add the missing keys to the `ThemeTokens` interface and `defaultTheme` in `tokens.ts`:
```ts
orange: '#f5a623',  // align with existing amber
blue:   '#3b82f6',
red:    '#ef4444',
```
Then emit them in `css-generator.ts` and remove the fallback hex values from all call sites: `color: var(--orange)` (no fallback needed once the token exists).

### BUG-02 · Fallback hex values don't match the corresponding design tokens
**Severity:** HIGH
**Files:**
- `src/lib/playgrounds/bank-reconciliation/components/ReconciliationStatement.svelte:275` — `var(--green, #16a34a)`
- `src/lib/playgrounds/bank-reconciliation/components/BankStatementPanel.svelte:218,222` — `var(--accent, #6ea8fe)`
- `src/lib/playgrounds/bank-reconciliation/components/TransactionJournal.svelte:271` — `var(--green, #22c55e)`
- `src/lib/playgrounds/amortization/components/ScheduleTable.svelte:265` — `var(--green, #22c55e)`
- Multiple — `var(--error, #ef4444)`

**Explanation:** Even for tokens that _do_ exist, fallback hex values diverge from `defaultTheme`. Examples:
- `--green` token = `#34d399`; fallbacks use both `#22c55e` and `#16a34a` (three different greens in one playground).
- `--accent` token = `#7c7fff`; fallback is `#6ea8fe` (a completely different shade).
- `--error` token = `#f06070`; fallback is `#ef4444`.

A CSS var fallback is a failure path. If these colors appear in any environment where the token fails to inject (e.g., iframe context, partial hydration, future SSR change), the UI renders the wrong brand color with no warning.

**Suggested fix:** Audit all `var(--token, #hex)` calls. For tokens that exist, delete the hex fallback. For tokens that don't exist, add them per BUG-01. Both fixes converge on zero bare hex values outside `tokens.ts`.

### BUG-03 · `color: #fff` hardcoded in four non-theme files
**Severity:** MEDIUM
**Files:**
- `src/lib/playgrounds/journal-entry/components/EntryHistory.svelte:400` — `.btn-confirm` text color
- `src/routes/[lang]/chart-of-accounts/+page.svelte:423` — class badge text
- `src/routes/[lang]/chart-of-accounts/+page.svelte:458` — active chip text
- `src/routes/[lang]/learn/+page.svelte:202` — section tag text

**Explanation:** These are hardcoded `#fff` values outside the theme file, in violation of the CLAUDE.md convention ("all colors via CSS custom properties from theme tokens — NEVER hardcode hex colors"). While white-on-dark is currently correct in the midnight theme, a future theme variant could render these elements unreadable.

**Suggested fix:** Add a `white: '#ffffff'` token to `defaultTheme` (or reuse `textPrimary` where semantically correct) and replace each instance with `color: var(--white)` or `color: var(--text-primary)`.

### BUG-04 · `account-link` button in EntryHistory has no accessible label
**Severity:** LOW
**File:** `src/lib/playgrounds/journal-entry/components/EntryHistory.svelte:117–126`

**Explanation:** The button's visible text is `{getAccountDisplay(line.accountKey)}`, which returns a string like `"401 - Fournisseurs, dettes en compte"`. This is typically sufficient for screen readers. However, the button sits inside a row of a table that already has a click handler (`onSelect` / `toggleExpand`), and the parent `<li>` has no `aria-label` describing the row's purpose. A screen reader user sees nested clickable elements with no disambiguation between "expand entry" and "jump to account". Adding `aria-label={$t('je.history.jumpToAccount', { account: getAccountDisplay(line.accountKey) })}` would disambiguate.

**Suggested fix:** Add an explicit `aria-label` to the inner account-link button using an i18n key that provides context (`"Navigate to account {name}"`).

### BUG-05 · `isFullscreen` state not reset on navigation away from playground
**Severity:** LOW
**File:** `src/routes/[lang]/playgrounds/[slug]/+page.svelte:43–52`

**Explanation:** `isFullscreen = $state(false)` is reset on mount, but if the user enters fullscreen and then navigates to another playground via SvelteKit's client-side routing (without a full page reload), the `fullscreenchange` event listener is cleaned up correctly, but `document.fullscreenElement` may still be non-null. The fullscreen state in the DOM and the component state can diverge. The `$effect` cleanup at line 56–62 handles the listener removal but does not call `document.exitFullscreen()` on teardown.

**Suggested fix:** In the `$effect` cleanup, add `if (document.fullscreenElement) { document.exitFullscreen(); }` before removing the listener.

### BUG-06 · `var(--accent, #6ea8fe)` fallback diverges badly in `color-mix()` expressions
**Severity:** MEDIUM
**File:** `src/lib/playgrounds/bank-reconciliation/components/TransactionJournal.svelte:394,405`

**Explanation:** Two rules use `background: var(--accent, #6ea8fe)` and `background: color-mix(in srgb, var(--accent, #6ea8fe) 88%, white)`. The `color-mix()` form additionally hardcodes the word `white` (an unresolved CSS keyword in some older engines — should be `#fff` or a token). When `--accent` resolves correctly these expressions work, but the fallback path produces a visually different tint than the rest of the accent system. See BUG-02 for root cause.

**Suggested fix:** Remove the fallback from `var(--accent)` (the token always resolves when the theme injector runs). Replace `white` in `color-mix()` with `transparent` or an explicit hex, or with `var(--bg)` for a surface-relative mix.

### IDEA-01 · Working Capital Simulator / Simulateur de BFR

**Title EN:** Working Capital & Cash Cycle Simulator
**Title FR:** Simulateur du Besoin en Fonds de Roulement (BFR)
**Target learner:** Licence 2–3, Master 1
**Pedagogical objective:** Students routinely confuse profit with cash. This playground makes the cash conversion cycle tangible: they see that a profitable firm can still run out of cash if receivables are long and payables are short.
**Core interaction:** Three sliders — Days Sales Outstanding (DSO), Days Inventory Outstanding (DIO), Days Payable Outstanding (DPO). A live chart shows the cash gap (BFR = DSO + DIO − DPO) and a timeline of cash in/out over a 12-month operating cycle. A "growth shock" button doubles sales volume to show how a fast-growing business can be cash-negative.
**Minimum viable inputs:** Annual revenue, COGS ratio, DSO (0–120 days), DIO (0–90 days), DPO (0–90 days).
**Outputs:** BFR in currency, cash gap in days, funding need, animated cash cycle diagram.
**Why it complements existing playgrounds:** TVM covers time value; bank reconciliation covers bank-vs-ledger gaps. BFR covers the operational cash cycle — a distinct concept that SYSCOHADA financial analysis (Tableau de Financement) depends on. Connects directly to working capital line items on the SYSCOHADA balance sheet (classes 3, 4, 5).

### IDEA-02 · VAT Return Builder / Déclaration TVA OHADA

**Title EN:** VAT Computation & Return Simulator
**Title FR:** Simulateur de Déclaration TVA (Zone OHADA)
**Target learner:** Licence 3, Master, practicing professional
**Pedagogical objective:** VAT mechanics (input credit, output tax, net payable vs refund position) are universally misunderstood by students who confuse VAT collected with firm revenue. OHADA-zone rates vary by country and product class — a common exam trap.
**Core interaction:** Learner enters a series of purchase and sale invoices (with amounts HT, TVA rate, deductibility flag). The playground computes input VAT, output VAT, net position, and renders a draft TVA return form in the SYSCOHADA layout. A country selector (Cameroun 19.25 %, Sénégal 18 %, Côte d'Ivoire 18 %, CEMAC standard) adjusts applicable rates.
**Minimum viable inputs:** List of invoices (type: purchase/sale, amount excl. tax, TVA rate, fully/partially deductible). Country selector.
**Outputs:** Input VAT subtotal, output VAT subtotal, net to pay or refund, filled return summary, per-line status (deductible / excluded / exempt).
**Why it complements existing playgrounds:** The journal-entry playground handles the accounting entries for VAT; this playground focuses on the fiscal computation that generates those entries — a natural upstream companion.

### IDEA-03 · Variance Analysis Workbench / Atelier d'Analyse des Écarts

**Title EN:** Standard Costing & Variance Analysis
**Title FR:** Coûts Standards et Analyse des Écarts
**Target learner:** Licence 3, Master 1 (Management Accounting / Contrôle de Gestion)
**Pedagogical objective:** Variance analysis (price, volume, mix, yield) is the most calculation-dense topic in SYSCOHADA management accounting. Students lose track of which variances are favourable vs unfavourable and which managers are responsible. A visual decomposition tree fixes this.
**Core interaction:** Learner enters standard cost card (standard qty × standard price per unit of output) and actual results (actual qty × actual price, actual output). The playground decomposes the total cost variance into a Price Variance and a Usage/Efficiency Variance, rendered as a waterfall bar chart. A "drill down" panel shows the decomposition formula step by step with sign conventions.
**Minimum viable inputs:** Standard price, standard quantity per unit, actual price, actual quantity used, actual units produced.
**Outputs:** Total variance, price variance, usage variance, favourable/unfavourable labels, responsibility assignment (purchasing vs production), waterfall chart.
**Why it complements existing playgrounds:** CVP covers volume decisions at the planning stage; variance analysis covers the post-period control stage. Together they span the full management accounting cycle. No existing playground covers cost control.

### IDEA-04 · Indirect Cash Flow Statement Builder / Tableau des Flux de Trésorerie (Méthode Indirecte)

**Title EN:** Cash Flow Statement — Indirect Method
**Title FR:** Tableau des Flux de Trésorerie — Méthode Indirecte (SYSCOHADA)
**Target learner:** Licence 3, Master
**Pedagogical objective:** SYSCOHADA mandates the indirect method for the cash flow statement (Tableau des Flux de Trésorerie). Students confuse the adjustments (add back depreciation, subtract increase in receivables, etc.) and produce sign errors on every exercise. An interactive builder that enforces the SYSCOHADA classification removes the sign-confusion by making the direction of each adjustment self-evident.
**Core interaction:** The learner is given a simplified income statement and comparative balance sheets. They drag-and-drop or classify each balance sheet movement (increase/decrease in each line item) into operating / investing / financing buckets, then see the reconstructed cash flow statement update live. A "reveal" button shows the correct classification for any line the learner got wrong.
**Minimum viable inputs:** Net income, depreciation charge, change in receivables, change in inventories, change in payables, capex, loan proceeds/repayments, dividends paid.
**Outputs:** Cash from operations, investing, financing; net change in cash; reconciliation to opening/closing cash balance; SYSCOHADA-formatted statement PDF export (stretch goal).
**Why it complements existing playgrounds:** The journal-entry playground generates the accounting entries; the BFR playground (IDEA-01) shows the operating cycle. The cash flow statement is the synthesis document — the third mandatory SYSCOHADA financial statement after the balance sheet and income statement, and currently absent from the playground catalog.

### IDEA-05 · Financial Ratios Dashboard / Tableau de Bord des Ratios Financiers

**Title EN:** SYSCOHADA Financial Ratio Dashboard
**Title FR:** Tableau de Bord des Ratios Financiers SYSCOHADA
**Target learner:** Licence 2–3, Master, practicing professional
**Pedagogical objective:** Students can compute individual ratios but rarely understand how they relate — e.g., that improving inventory turnover improves both the liquidity ratio and the ROE via the DuPont decomposition. A connected dashboard makes these relationships visible simultaneously.
**Core interaction:** Learner enters a simplified SYSCOHADA balance sheet (classes 1–5) and income statement (classes 6–7). The dashboard instantly computes 12 key ratios grouped into four families: Liquidity (current ratio, quick ratio, cash ratio), Solvency (debt-to-equity, gearing, interest coverage), Profitability (gross margin, net margin, ROA, ROE), and Activity (asset turnover, receivables days). Changing any input live-updates all 12 ratios. A DuPont decomposition tree shows how ROE decomposes into margin × turnover × leverage.
**Minimum viable inputs:** Total current assets, cash, receivables, inventories, total assets, total equity, total debt, current liabilities, revenue, EBIT, net income, interest expense.
**Outputs:** 12 computed ratios with colour-coded benchmark bands (green = healthy, amber = watch, red = distress), DuPont tree, radar chart of ratio families.
**Why it complements existing playgrounds:** All other playgrounds teach how to produce individual financial figures; this one teaches how to read and interpret the full picture. It is the analytical endpoint of the accounting curriculum and the most frequently tested item in professional certification exams (DSCG, OHADA expert-comptable).
