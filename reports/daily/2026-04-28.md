### Issue 1 — Privacy page: all user-facing strings are hardcoded English, `t` import is unused dead code
**Severity:** HIGH  
**File:** `src/routes/[lang]/privacy/+page.svelte:1-27`

Every user-visible string — `<title>`, `<h1>`, five `<h2>` headings, and four `<p>` paragraphs — is hardcoded English with no French equivalent. A French user visiting `/fr/privacy` sees English content, violating the EN/FR parity requirement in CLAUDE.md.

Remove the dead `import { t }` and replace with `import { t as $t } from '$lib/i18n'` used properly.

### Issue 2 — Three undefined CSS variables (`--orange`, `--blue`, `--red`) cause fallback hex colors to always activate
**Severity:** HIGH  
**Files (representative sample):**
- `src/lib/components/playground/KpiStrip.svelte:76` — `color: var(--orange, #f59e0b)`
- `src/lib/playgrounds/amortization/components/ScheduleTable.svelte:240,246,259,265`
- `src/lib/playgrounds/amortization/components/ChartPanel.svelte:436,444,448,493,501,505,515,529,533`
- `src/lib/playgrounds/amortization/components/InputsPanel.svelte:604,605`
- `src/lib/playgrounds/bank-reconciliation/components/ReconciliationStatement.svelte:275,279,292,293,297,298`
- `src/lib/playgrounds/bank-reconciliation/components/VarianceScale.svelte:223,224,227,230` (and ~10 more lines)
- `src/lib/playgrounds/bank-reconciliation/components/LedgerPanel.svelte:149,175,208,209,210`
- `src/lib/playgrounds/bank-reconciliation/components/UnmatchedItemsPanel.svelte:188`

**Explanation:** `src/lib/theme/css-generator.ts` emits `--green`, `--amber`, `--error`, and `--accent`, but never `--orange`, `--blue`, or `--red`. Every component that writes `var(--orange, #f59e0b)` silently falls through to the raw hex. Worse, the fallbacks are inconsistent with the actual theme palette: `--amber` is `#f5a623` (not `#f59e0b`), `--error` is `#f06070` (not `#ef4444`), `--accent` is `#7c7fff` (not `#3b82f6`). Even `var(--green, #22c55e)` uses the wrong green (#34d399 in tokens). The entire color system is silently ignored for these components: switching themes would have no effect.  
**Suggested fix:** In `css-generator.ts`, add `--orange: ${c.amber};` (or add a proper `orange` token to `ThemeTokens`), `--blue: ${c.accent};`, and `--red: ${c.error};` — or replace each `var(--orange, ...)` with the correct canonical variable (`--amber`, `--accent`, `--error`). The latter is cleaner and avoids alias proliferation.

### Issue 3 — Four instances of `color: #fff` bypassing the theme system entirely
**Severity:** MEDIUM  
**Files:**
- `src/lib/playgrounds/journal-entry/components/EntryHistory.svelte:400` — `.btn-confirm { color: #fff; }`
- `src/routes/[lang]/chart-of-accounts/+page.svelte:423` — `.framework-btn.active { color: #fff; }`
- `src/routes/[lang]/chart-of-accounts/+page.svelte:458` — `.class-chip.active { color: #fff; }`
- `src/routes/[lang]/learn/+page.svelte:202` — `.group-cta { color: #fff; }`

**Explanation:** All four cases apply white text on an accent-colored button/chip background. There is no `--text-on-accent` token in `tokens.ts`, so authors used `#fff` directly. If the accent color were ever changed to a light hue, white text would become invisible. This is a hardcoded value that bypasses the design token system in violation of CLAUDE.md.  
**Suggested fix:** Add a `textOnAccent: '#ffffff'` field to `ThemeTokens.colors`, emit `--text-on-accent: ${c.textOnAccent};` in the CSS generator, and replace all four `#fff` instances with `var(--text-on-accent)`.

### Issue 4 — Chart color maps in bank-reconciliation components are hardcoded JS strings, not theme-aware
**Severity:** MEDIUM  
**Files:**
- `src/lib/playgrounds/bank-reconciliation/components/CategoryBreakdownDonut.svelte:30-39` — `CATEGORY_COLORS` object with 9 hex strings
- `src/lib/playgrounds/bank-reconciliation/components/MatchingPairsOverlay.svelte:31-34` — `MATCH_COLORS` with 4 hex strings

**Explanation:** These colors are passed programmatically as SVG `stroke`/`fill` attributes, so CSS variables cannot be used directly in the attribute value. However, the values are hardcoded strings scattered across two files with no single source of truth. If the app grows to support a light theme or accessible high-contrast mode, these 13 values would need to be hunted down and changed manually. Additionally, the `#6ea8fe` in `CategoryBreakdownDonut` does not match `--accent` (`#7c7fff`), suggesting drift has already occurred.  
**Suggested fix:** Create `src/lib/theme/chart-colors.ts` that exports typed color maps (e.g. `CATEGORY_PALETTE`, `MATCH_PALETTE`). Values can be read from CSS custom properties at runtime via `getComputedStyle(document.documentElement).getPropertyValue('--accent')` or simply kept as named constants in one place. Both components import from the shared module instead of defining their own.

### Idea 1 — Financial Ratio Analysis
**Title (EN):** Financial Ratio Dashboard  
**Title (FR):** Tableau de Bord des Ratios Financiers  
**Target learner:** Licence 3, Master, practicing professionals  
**Pedagogical objective:** Students can memorize ratio formulas without understanding which SYSCOHADA account codes feed each ratio, or how ratios interact. This playground makes the mapping from SYSCOHADA account classes (1–5) to ratios explicit and visible.  
**Core interaction:** Learner is given a pre-populated BILAN/COMPTE DE RÉSULTAT in SYSCOHADA format. Sliders adjust key account balances (e.g. Class 4 receivables, Class 5 cash). Six ratio panels update live: current ratio, quick ratio, debt-to-equity, asset turnover, net margin, return on equity. Color-coded bands show healthy/warning/critical zones per OHADA sector norms.  
**Minimum viable inputs:** Class 1–5 account balances (8–10 sliders); output: 6 ratio cards + an interpretation badge per ratio.  
**Why it complements existing playgrounds:** The journal-entry playground stops at the trial balance level. This playground picks up from the financial statements already generated, forming a natural learning progression: journal → statements → ratios → decisions.

### Idea 2 — TVA (VAT) in the OHADA Zone
**Title (EN):** OHADA VAT Calculator  
**Title (FR):** Calculateur de TVA — Zone OHADA  
**Target learner:** Licence 2, practicing professionals (accountants, tax preparers)  
**Pedagogical objective:** Students frequently confuse TVA collectée (output tax, SYSCOHADA account 4431/4435), TVA déductible (input tax, 4452/4456), and the resulting TVA à décaisser or crédit de TVA. Rate variation across OHADA states (Cameroon 19.25%, Côte d'Ivoire 18%, Senegal 18%, Burkina 18%, etc.) adds another confusion layer.  
**Core interaction:** Learner selects country (dropdown populates the applicable rate), enters a set of sales invoices and purchase invoices for the month. The tool computes TVA collectée, TVA déductible, and the net position. A T-account view maps each entry to the correct SYSCOHADA account. A "declaration summary" panel mimics the actual monthly VAT return layout used in francophone Africa.  
**Minimum viable inputs:** Country selector, N sales lines (amount HT, taxable Y/N), N purchase lines (amount HT, deductible Y/N); outputs: TVA collectée, TVA déductible, solde, SYSCOHADA journal entries, declaration summary.  
**Why it complements existing playgrounds:** No existing playground touches indirect taxes or country-level regulatory variation. This fills a major curriculum gap unique to OHADA-zone students vs. generic accounting courses.

### Idea 3 — Working Capital & Cash Gap (BFR)
**Title (EN):** Working Capital & Cash Gap  
**Title (FR):** Fonds de Roulement et BFR  
**Target learner:** Licence 3, Master  
**Pedagogical objective:** The Besoin en Fonds de Roulement (BFR) is one of the most tested concepts in SYSCOHADA financial analysis exams, yet students consistently confuse FR (fonds de roulement net global), BFR (operating), BFRHE (non-operating), and TN (net treasury). The misconception is treating BFR as a single number rather than a structural decomposition.  
**Core interaction:** A balance sheet scaffold pre-filled with SYSCOHADA account class labels. The learner drags sliders for Class 3 (stocks), Class 4 receivables, Class 4 payables, Class 5 (cash). Three live panels compute FR = (capitaux permanents − actif immobilisé), BFR = (actif circulant d'exploitation − passif circulant d'exploitation), TN = FR − BFR. A waterfall chart visualizes the decomposition. Toggle between "surplus BFR financed by cash" vs. "TN negative → short-term borrowing risk" scenarios.  
**Minimum viable inputs:** 6–8 balance sheet line sliders; outputs: FR, BFR, BFRHE, TN calculations + waterfall chart + risk interpretation badge.  
**Why it complements existing playgrounds:** The TVM playground models time-based cash flows; this playground models structural cash position — complementary dimensions of financial analysis.

### Idea 4 — Provisions & Impairment Calculator
**Title (EN):** Provisions & Impairment  
**Title (FR):** Provisions et Dépréciations  
**Target learner:** Licence 3, Master  
**Pedagogical objective:** Students conflate provisions for risks/charges (SYSCOHADA Class 1: accounts 15x) with depreciation of assets (Class 2: accounts 29x) and impairment of receivables (Class 4: accounts 49x). Each has distinct booking patterns, tax treatment, and reversal conditions under OHADA GAAP.  
**Core interaction:** Three tabs — Provisions pour risques (15x), Dépréciations sur immobilisations (29x), Dépréciations sur créances (49x). Each tab presents a scenario (e.g. "pending lawsuit, estimated loss 2 000 000 FCFA"). The learner selects the account, enters the amount, and the tool generates the journal entry and shows the balance sheet/P&L impact. A "reversal" toggle shows the reverse entry when the risk materializes vs. when it dissipates.  
**Minimum viable inputs:** Type selector (risk/impairment), amount, probability (low/medium/certain); outputs: journal entries for booking and reversal, P&L impact, balance sheet presentation.  
**Why it complements existing playgrounds:** The journal-entry playground focuses on routine transactions. Provisions involve judgment and estimation — a distinct skill tested at Licence 3 and in professional exams (DSCG equivalent).

### Idea 5 — Break-Even with SYSCOHADA Cost Classification
**Title (EN):** SYSCOHADA Cost Classification & Margin Analysis  
**Title (FR):** Charges par Nature et Analyse des Marges  
**Target learner:** Licence 2, Licence 3  
**Pedagogical objective:** The existing CVP playground uses generic "fixed cost / variable cost" inputs. SYSCOHADA's chart of accounts organizes costs by nature (Class 6: 60x materials, 61x services, 62x transportation, 63x taxes, 64x personnel, 65x other, 66x finance, 68x depreciation). Students must learn to reclassify nature-based costs into the taux de variabilité framework before computing the contribution margin.  
**Core interaction:** A pre-filled income statement with SYSCOHADA Class 6 account lines and amounts. Learner assigns a variability coefficient (0–100%) to each cost line. The tool computes: charges variables totales, charges fixes totales, marge sur coût variable (MCV), taux de MCV, seuil de rentabilité (break-even), point mort (in days), and marge de sécurité. A live chart shows the CVP diagram with SYSCOHADA-native labeling.  
**Minimum viable inputs:** Revenue, 6–8 Class 6 cost lines with variability slider each; outputs: MCV, taux MCV, seuil de rentabilité, point mort, marge de sécurité, CVP chart.  
**Why it complements existing playgrounds:** The existing CVP playground is an excellent generic tool but lacks SYSCOHADA vocabulary and cost-nature classification, which is precisely what OHADA-zone exams test. This playground acts as the SYSCOHADA-native counterpart rather than a duplicate.
