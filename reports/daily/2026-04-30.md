### [HIGH] Undefined CSS custom properties — `--orange`, `--blue`, `--red` always fall back to hardcoded hex

**Severity:** High
**Files:**
- `src/lib/components/playground/KpiStrip.svelte:76`
- `src/lib/playgrounds/amortization/components/ScheduleTable.svelte:240,246,259`
- `src/lib/playgrounds/amortization/components/ChartPanel.svelte:436,444,448,493,501,505,515,529,533`
- `src/lib/playgrounds/amortization/components/InputsPanel.svelte:604–605`
- `src/lib/playgrounds/bank-reconciliation/components/LedgerPanel.svelte:175`
- `src/lib/playgrounds/bank-reconciliation/components/BankStatementPanel.svelte:218–222`
- `src/lib/playgrounds/bank-reconciliation/components/TransactionJournal.svelte:382`

**Explanation:** `var(--orange, #f59e0b)`, `var(--blue, #3b82f6)`, and `var(--red, #ef4444)` are used widely across amortization and bank-reconciliation components. None of these three CSS variables are emitted by `css-generator.ts` (which only produces `--amber`, `--green`, `--error`, and their variants). Because the property never resolves, the hardcoded fallback always applies — making these components permanently bypass the theme system. The codebase uses `--amber` for warm orange tones, `--error` for red; there is no blue semantic token at all.

**Suggested fix:** Add `--orange` and `--blue` to `ThemeTokens`, `defaultTheme`, and `css-generator.ts`, mapping to semantically appropriate values (`amber` for orange, a new `info` color for blue). Alternatively, replace `var(--orange, ...)` with `var(--amber)` and introduce an `info` token for chart series that currently rely on a blue.

### [HIGH] Wrong fallback hex values diverge from actual theme colors

**Severity:** High
**Files:**

**Explanation:** `var(--green, #16a34a)` and `var(--error, #ef4444)` are used as fallbacks. The actual theme tokens are `green: '#34d399'` and `error: '#f06070'` — neither matches the fallback. If CSS var resolution ever fails (e.g., SSR flash before hydration, browser quirk), the rendered color will be visually wrong and inconsistent with the design system.

**Suggested fix:** Remove all fallback values from `var(--token, fallback)` usage; rely on the theme injection. If a fallback is required for resilience, it must match the token value in `tokens.ts`.

### [MEDIUM] `requestFullscreen()` / `exitFullscreen()` unguarded — can throw `DOMException`

**Severity:** Medium
**File:** `src/routes/[lang]/playgrounds/[slug]/+page.svelte:47,50`

**Explanation:** Calling `target.requestFullscreen()` throws a `DOMException` if the user denies the permission, the document is sandboxed, or the API is unsupported. `document.exitFullscreen()` similarly throws when no element is fullscreen. Neither call is wrapped in `try/catch`, so an unhandled rejection will surface in the console and may break `isFullscreen` state.

**Suggested fix:**
```ts
async function toggleFullscreen() {
  const target = data.isNativeModule ? nativeContainerEl : iframeEl;
  if (!target) return;
  try {
    if (!document.fullscreenElement) {
      await target.requestFullscreen();
      isFullscreen = true;
    } else {
      await document.exitFullscreen();
      isFullscreen = false;
    }
  } catch {
    /* fullscreen denied or unavailable */
  }
}
```

**Severity:** Medium
**Files:**

**Suggested fix:** Replace all `rgba(255,255,255,0.02)` with `color-mix(in srgb, var(--text-primary) 2%, transparent)`.

**Severity:** Low (by design for Phase 1)
**File:** `src/routes/api/feedback/+server.ts`

**Explanation:** The rate-limiter uses a module-level `Map<string, { count; resetAt }>`. ### Idea 1 — VAT Ledger Reconstructor
**EN:** VAT Ledger Reconstructor | **FR:** Reconstructeur de comptes TVA

**Target learner:** Licence 2

**Pedagogical objective:** Students confuse TVA collectée (output VAT) with TVA déductible (input VAT), and miscalculate the net TVA payable. This playground forces learners to trace VAT through a sequence of purchase and sale transactions, which mirrors real monthly VAT return preparation under SYSCOHADA.

**Core interaction:** The learner inputs a sequence of purchase and sale invoices (amount HT, applicable VAT rate). The playground reconstructs the TVA collectée, TVA déductible, and TVA à décaisser ledger accounts, then generates the monthly VAT return summary (formulaire de déclaration TVA simplifié).

**Minimum viable inputs / outputs:**
- Inputs: list of transactions (type: achat/vente, amount HT, rate: 0/10/18/20%), reporting month
- Outputs: TVA collectée total, TVA déductible total, net TVA payable or credit carryforward, journal entries for accounts 4431 and 4455 (SYSCOHADA plan comptable)

**Why it complements existing playgrounds:** Journal Entry covers debits and credits in the abstract; this focuses on the specific VAT mechanics that students encounter on every real invoice.

### Idea 2 — Working Capital Diagnosis (FR / BFR / TN)
**EN:** Working Capital Diagnosis | **FR:** Diagnostic Fonds de Roulement / BFR / Trésorerie

**Target learner:** Licence 3 / Master 1

**Pedagogical objective:** The FR–BFR–TN triangle is the cornerstone of SYSCOHADA financial diagnosis, but students routinely mix up what belongs in permanent capital versus operating cycle. This playground makes the classification decision explicit and immediate.

**Core interaction:** The learner inputs a simplified SYSCOHADA balance sheet (capitaux permanents, actif immobilisé, actif circulant, dettes circulantes, trésorerie actif/passif). The playground computes FR = Capitaux permanents − Actif immobilisé, BFR = (Actif circulant − Trésorerie actif) − (Dettes circulantes − Trésorerie passif), TN = FR − BFR, and renders a color-coded waterfall chart showing the relationship.

**Minimum viable inputs / outputs:**
- Inputs: six balance-sheet buckets (numeric)
- Outputs: FR, BFR, TN with sign interpretation (equilibrium, tension, surplus), waterfall visualization, plain-language diagnosis sentence in EN/FR

**Why it complements existing playgrounds:** Amortization covers individual liability schedules; this covers whole-company financial health using standard OHADA balance-sheet structure.

### Idea 3 — Inventory Valuation Methods (CUMP / FIFO)
**EN:** Inventory Valuation Methods | **FR:** Valorisation des stocks — CUMP et FIFO

**Target learner:** Licence 2 / Licence 3

**Pedagogical objective:** SYSCOHADA mandates the weighted average cost (CUMP) method; IFRS allows FIFO. Students struggle to recompute CUMP after each receipt and to see how method choice affects cost of goods sold and ending inventory valuation.

**Core interaction:** The learner builds a stock card (fiche de stock) by adding a sequence of receipts (entrées) and issues (sorties) with quantities and unit prices. The playground recomputes the CUMP after each receipt and values each issue under both CUMP and FIFO, side-by-side. A summary table shows ending stock value and cumulative COGS under each method.

**Minimum viable inputs / outputs:**
- Inputs: opening balance, sequence of receipts/issues (date, quantity, unit cost for receipts)
- Outputs: line-by-line fiche de stock (CUMP column, FIFO column), ending stock value, total COGS, variance between methods

**Why it complements existing playgrounds:** Depreciation handles asset value erosion over time; this handles the distinct (and examined) topic of current-asset inventory valuation.

### Idea 4 — SYSCOHADA Financial Ratios Dashboard
**EN:** Financial Ratios Dashboard | **FR:** Tableau de bord des ratios financiers SYSCOHADA

**Target learner:** Licence 3 / Master 1 / practicing professional

**Pedagogical objective:** Financial ratio interpretation is tested heavily in DSCG/expertise comptable exams, but students lack a tool that maps ratio formulas directly to the SYSCOHADA balance sheet and income statement line numbers. They often apply Franco-French ratios that don't match the OHADA chart structure.

**Core interaction:** The learner inputs six to ten key SYSCOHADA financial statement line items (actif immobilisé net, actif circulant, dettes financières, chiffre d'affaires, EBE, résultat net). The playground computes a dashboard of twelve ratios (solvabilité générale, autonomie financière, rentabilité économique, rentabilité financière, délai de rotation stocks, etc.) with a color-coded gauge for each ratio, SYSCOHADA-specific formulas, and a sector benchmark comparison selector.

**Minimum viable inputs / outputs:**
- Inputs: ~10 SYSCOHADA line items, sector selector (commerce, industrie, services)
- Outputs: 12 computed ratios with gauges, reference ranges, one-sentence interpretation each

**Why it complements existing playgrounds:** Complements the FR/BFR playground (Idea 2) at a higher analytical level. Distinct from CVP (managerial) and TVM (financial math) — this is pure financial diagnosis.

### Idea 5 — Payroll & Social Charges Calculator (Zone OHADA)
**EN:** Payroll & Social Charges Calculator | **FR:** Calculateur de bulletin de paie — Zone OHADA

**Target learner:** Licence 2 / practicing professional

**Pedagogical objective:** Gross-to-net salary calculation with the CNPS (Caisse Nationale de Prévoyance Sociale), IRPP (Impôt sur le Revenu des Personnes Physiques), and other statutory deductions is country-specific within the OHADA zone and rarely covered by generic accounting simulators. Students confuse employer charges with employee charges.

**Core interaction:** The learner selects a country (Cameroun, Côte d'Ivoire, Sénégal, Gabon — initial subset), inputs gross salary, number of dependants, and any taxable benefits in kind. The playground computes statutory deductions (employee CNPS, IRPP progressive scale, CFCE, employer charges), produces a bulletin de paie layout, and shows the employer total cost vs. net take-home.

**Minimum viable inputs / outputs:**
- Inputs: country, gross salary, dependants, benefits in kind
- Outputs: structured bulletin de paie with each deduction line, employer vs. employee split, net salary, journal entries booking salary expense (accounts 661x SYSCOHADA)

**Why it complements existing playgrounds:** No existing playground touches payroll. It directly links to Journal Entry concepts (posting salary expense) while covering practical OHADA-zone tax law that is examined in Licence programmes.
